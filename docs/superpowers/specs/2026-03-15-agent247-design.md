# agent247 — Local LLM Agent Task Scheduler

A local daemon framework that runs Claude-powered developer tasks on a cron schedule, tracks results, and waits for human review before final action.

## Problem

Developer maintenance tasks (reviewing dependabot PRs, triaging review requests, etc.) are repetitive and time-consuming. Running Claude agents on a schedule can handle the analysis, but there's no framework to manage scheduling, deduplication, reporting, and review workflow.

## Core Concepts

- **Task**: A folder under `tasks/` with `config.yaml` and `prompt.md`. Defines what to do, when, and how to discover work items.
- **Run**: A single execution of a task, stored under `runs/<ulid>/`. Contains metadata, logs, rendered prompt, raw Claude output, and a human-readable report.
- **Work Item**: An individual unit of work (e.g. a specific PR) identified by a key (e.g. URL). Deduplication prevents re-processing items that already have active runs.

## Tech Stack

- **Language**: TypeScript (Node.js)
- **CLI framework**: commander
- **Config format**: YAML
- **IDs**: ULID (lexicographically sortable, timestamp + randomness)
- **Storage**: Filesystem (YAML + Markdown files)
- **Scheduling**: System crontab (fenced section)

## Task Config Format

```yaml
# tasks/review-dependabot/config.yaml
name: Review Dependabot PRs
schedule: "*/30 * * * *"
timeout: 300                    # seconds
enabled: true
vars:                           # task-level variables (override global vars.yaml)
  repo_owner: noootown
discovery:
  command: "gh pr list --author 'dependabot[bot]' --json url,number,title"
  item_key: url                 # field used for dedup
prompt_mode: per_item           # per_item | batch
lifecycle:
  auto_resolve: true
  resolve_command: "gh pr view {{url}} --json state -q '.state'"
  resolve_when: "MERGED|CLOSED"
```

### Fields

- `name`: Human-readable task name
- `schedule`: Cron expression
- `timeout`: Max seconds before killing the claude process
- `enabled`: Whether the task is active (synced to crontab)
- `vars`: Optional task-level variables (override global vars.yaml, overridden by item fields)
- `discovery.command`: Shell command that returns JSON array of work items
- `discovery.item_key`: Field in each item used as the dedup key
- `prompt_mode`: `per_item` runs the prompt once per new item; `batch` runs once with all items
- `lifecycle.auto_resolve`: Whether to auto-check if work items are resolved
- `lifecycle.resolve_command`: Shell command to check status (receives `{{url}}` and other item fields)
- `lifecycle.resolve_when`: Regex pattern — if resolve command output matches, mark as resolved

## Template Variables

```yaml
# vars.yaml
github_username: noootown
platform_repo_path: /Users/noootown/Downloads/projects/platform
```

Prompt files use `{{var_name}}` for substitution. The rendered prompt is saved to `runs/<ulid>/prompt.rendered.md` — never written back to the task folder.

### Variable Precedence

When resolving `{{var_name}}`, the order of precedence (highest wins):

1. **Item fields** — from discovery output (e.g. `url`, `number`, `title`)
2. **Task-level vars** — from `config.yaml` `vars:` section
3. **Global vars** — from `vars.yaml`

For `batch` mode, item fields are not available as individual variables. Instead, the full array is injected as `{{items_json}}` (JSON string) and `{{items_list}}` (markdown bullet list of `item_key` values).

## Environment Variables

Secrets and credentials are managed via environment files:

```bash
# dev.env — committed, serves as documentation of required vars
GITHUB_TOKEN=
SLACK_TOKEN=
# Add other secrets as needed
```

```bash
# .env.local — gitignored, actual values
GITHUB_TOKEN=ghp_xxxxxxxxxxxx
SLACK_TOKEN=xoxb-xxxxxxxxxxxx
```

Loading order: `dev.env` is loaded first (provides defaults/empty values), then `.env.local` overrides it. The runner loads both files before spawning any process. Environment variables are available to:

- **Discovery commands** (e.g. `gh` uses `GITHUB_TOKEN` automatically)
- **Lifecycle resolve commands**
- **Claude process** (passed through to the spawned `claude` process)

Environment variables are NOT available as `{{var}}` template variables in prompts — they are process-level env vars only. Use `vars.yaml` for prompt template substitution.

## Prompt Format

Prompts should instruct Claude to output a structured header for metadata extraction:

```markdown
<!-- tasks/review-dependabot/prompt.md -->
Review the following Dependabot PR: {{url}}

Repository: {{github_username}}'s repos

Provide:
- Summary of the dependency change
- Risk assessment (breaking change? major version bump?)
- Recommendation: merge / needs manual review / skip

Output format:
- First line: URL of the PR
- If there is nothing to do, respond with exactly: NO_ACTION
```

The runner extracts the URL from the first line and checks for `NO_ACTION` to determine run status.

## Run Lifecycle

1. Cron fires `agent247 run <task-id>`
2. **Lock**: Acquire a PID-file lock at `tasks/<task-id>/.lock`. If a lock exists and the PID is still alive, log a skip and exit (no run created). If the PID is stale (process dead), remove the lock and proceed.
3. **Lifecycle resolution**: Before discovery, run `resolve_command` on all unresolved `completed` and `error` runs for this task. Mark as `resolved` if output matches `resolve_when`. This ensures both successful and failed runs get cleaned up when the underlying work item is no longer relevant.
4. **Discovery**: Execute `discovery.command`, parse JSON array of work items. If the command fails (non-zero exit or malformed JSON), create an `error` run with the failure details in log.txt and exit. An empty array is a valid result (no items to process).
5. **Dedup**: Scan `runs/*/meta.yaml` for this task — find items with matching `item_key` that are `completed` or `error` and not `resolved`. Skip `completed` items. For `error` items, re-process them (errors are retried on next run).
6. **Execute or skip**: If no new/retriable items → write a `skipped` run (meta.yaml + log.txt only). Otherwise:
   - For `per_item`: run prompt once per new item, create a run per item
   - For `batch`: run prompt once with all new items, create a single run
7. **Render prompt**: Substitute variables into prompt.md (see Variable Precedence)
8. **Call Claude**: `claude -p "<rendered prompt>" --output-format json`
9. **Parse output**: Extract URL from response, detect `NO_ACTION`, capture full JSON
10. **Determine status**: If Claude responded with `NO_ACTION` → `no-action`. If meaningful output → `completed`. If process failed → `error`.
11. **Write artifacts**: meta.yaml, prompt.rendered.md, raw.json, report.md, log.txt. For `no-action` runs, all artifacts are still written (the Claude response is preserved). For `skipped` runs, only meta.yaml + log.txt.
12. **Release lock**: Remove the PID-file lock.

## Run Metadata

```yaml
# runs/<ulid>/meta.yaml
schema_version: 1
id: 01H5K3P8G0XQJN4R2BVTW9MC6E
task: review-dependabot
status: completed              # skipped | no-action | completed | error | resolved
reviewed: false
url: https://github.com/noootown/repo/pull/42
item_key: https://github.com/noootown/repo/pull/42
started_at: 2026-03-15T10:30:00Z
finished_at: 2026-03-15T10:31:42Z
duration_seconds: 102
exit_code: 0
```

### Run Statuses

- `skipped`: No new or retriable items discovered (dedup filtered everything out)
- `no-action`: Claude analyzed the item and determined nothing was needed (`NO_ACTION` response). Artifacts are still preserved.
- `completed`: Meaningful work done, awaiting review
- `error`: Discovery command failed, Claude process failed (timeout, crash, non-zero exit), or other runtime error. Error runs are retried on the next cron invocation.
- `resolved`: Work item is no longer relevant (PR merged/closed). Applied to both `completed` and `error` runs during lifecycle resolution.

## Run Artifacts

```
runs/<ulid>/
  meta.yaml              # run metadata and status
  prompt.rendered.md     # exact prompt sent to Claude (with vars substituted)
  raw.json               # full Claude output (--output-format json)
  report.md              # extracted final response (human-readable)
  log.txt                # framework-level execution log (timestamps, pid, duration)
```

For `skipped` runs, only `meta.yaml` and `log.txt` are written. For `no-action` runs, all artifacts are written (Claude's response is preserved for traceability).

## CLI Commands

### `agent247 sync`

Reads all `tasks/*/config.yaml` where `enabled: true`, generates crontab entries, and writes them into a fenced section of the user's crontab.

```crontab
# --- agent247 START ---
# review-dependabot (every 30 min)
*/30 * * * * /path/to/agent247 run review-dependabot >> /path/to/runs/cron.log 2>&1
# review-platform-prs (every 2 hours)
0 */2 * * * /path/to/agent247 run review-platform-prs >> /path/to/runs/cron.log 2>&1
# --- agent247 END ---
```

Preserves all crontab entries outside the fenced section.

### `agent247 list`

Lists all defined tasks with their schedule, enabled status, and last run time.

### `agent247 run <task-id>`

Executes a single task. Called by cron or manually. Handles the full discovery → dedup → execute → report flow.

### `agent247 status`

Shows unreviewed meaningful runs. Flags: `--all` (include no-ops), `--task <id>` (filter by task).

### `agent247 show <ulid>`

Displays the report for a specific run.

### `agent247 review <ulid>`

Marks a run as reviewed (sets `reviewed: true` in meta.yaml).

### `agent247 watch`

Persistent dashboard that auto-refreshes every 5 seconds. Shows unreviewed runs with clickable URLs. V1 implementation uses a simple clear-and-reprint loop (no TUI library needed). Interactive keybindings (`[r]` review, `[o]` open url, `[q]` quit) can be added in a later version if needed.

```
 agent247 — 3 unreviewed · 1 error · last run 2m ago        ↻ 5s

 STATUS   TASK                  TIME        URL
 ● new    review-dependabot     10:30       https://github.com/noootown/repo/pull/42
 ● new    review-platform-prs   10:00       https://github.com/noootown/repo/pull/108
 ✗ error  review-dependabot     09:30       —
```

### `agent247 clean`

Deletes old runs. Flags: `--older-than <duration>` (e.g. `7d`, `30d`), `--status <status>` (e.g. `skipped`, `no-action`). By default, only deletes reviewed runs. Use `--include-unreviewed` to also delete unreviewed runs.

## Concurrency Control

Each task uses a PID-file lock at `tasks/<task-id>/.lock` to prevent concurrent execution. The lock file contains the PID of the running process. On acquisition:

1. If no lock file exists → create it with current PID, proceed
2. If lock file exists and PID is alive → log "task already running, skipping" and exit cleanly (no run created)
3. If lock file exists and PID is dead → remove stale lock, create new one, proceed

The lock is always released in a `finally` block to ensure cleanup even on errors.

## Crontab Management

The sync command uses fenced markers (`# --- agent247 START ---` / `# --- agent247 END ---`) to manage only its own section of the crontab. Algorithm:

1. Read current crontab (`crontab -l`)
2. Find and remove everything between the markers (inclusive)
3. Generate new entries from enabled tasks
4. Append the fenced block
5. Write back (`crontab -`)

## Installation

The CLI will be built as a Node.js package and linked globally via `npm link` or installed to a path. The user will add it to their dotfiles for portability. The binary name is `agent247`.

## Initial Tasks (v1)

### review-dependabot
- Schedule: every 30 minutes
- Discovery: `gh pr list --author 'dependabot[bot]'`
- Mode: per_item
- Auto-resolve: when PR is merged or closed

### review-platform-prs
- Schedule: every 2 hours
- Discovery: `gh pr list --review-requested=@me` in platform repo
- Mode: per_item
- Uses the review-pr skill from `~/Downloads/projects/platform`
- Auto-resolve: when PR is merged or closed

## Out of Scope (v1)

- Multi-user support
- Remote/cloud execution
- Web UI
- SQLite or any database
- Version control of tasks or runs
- Explicit retry logic with backoff (error runs are retried on next cron invocation)
